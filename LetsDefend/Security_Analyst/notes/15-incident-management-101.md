# Incident Management 101

## What is Incident Management?
Incident Management is the process of identifying, investigating, responding to, documenting, and closing security incidents.

An **Incident Management System (IMS)** is the platform where SOC analysts manage alerts and conduct investigations.

**Examples**
- TheHive
- LetsDefend Case Management
- DFIE Iris (of which I have been using offlate)

---

## Key Definitions

### Event
Any observable activity on a system or network.

**Examples**
- User login
- Firewall blocks a connection
- Email sent
- File accessed

### Alert
A notification generated by a SIEM or security tool when suspicious activity matches a detection rule.

> An alert does **not** necessarily mean an incident has occurred.

### Incident
A confirmed or suspected violation of an organization's security policy that requires investigation and response.

---

## Alert Classification

### True Positive (TP)
The alert correctly identifies malicious activity.

**Example**
A SQL Injection detection rule triggers, and investigation confirms an actual SQL Injection attack.

### False Positive (FP)
The alert triggers, but investigation confirms there is no malicious activity.

**Example**
A search containing the word **"Union"** triggers an SQL Injection rule even though it is legitimate.

> False positives are common, and reducing them is an important part of SOC operations.

---

## How an Incident Management System Works

1. A security device or SIEM generates an alert.
2. The alert is sent to the Incident Management System.
3. A case/ticket is created.
4. The case is enriched using Threat Intelligence and SOAR integrations.
5. The analyst investigates the alert using the assigned playbook.
6. The analyst determines whether the alert is a True Positive or False Positive.
7. Findings are documented and the case is closed.

---

## IMS Integrations

### Threat Intelligence
Provides reputation and context for indicators such as:
- IP addresses
- Domains
- URLs
- File hashes

Examples:
- VirusTotal
- AbuseIPDB

### SOAR (Security Orchestration, Automation & Response)

Automates repetitive response actions such as:
- Blocking IPs
- Blocking domains
- Disabling user accounts
- Isolating endpoints

This speeds up incident response and reduces manual effort.

---

## Case Naming

Cases should have clear and consistent names.

**Example**

```
EventID: 123 - [Suspicious PowerShell Execution]
```

Consistent naming improves searching, reporting, and historical investigations.

---

## Playbooks

A **playbook** is a documented investigation procedure for a specific alert type.

Examples:
- Phishing
- Malware
- Ransomware
- Web attacks

Playbooks help analysts:
- Follow a consistent investigation process
- Reduce mistakes
- Standardize investigations across the SOC
- Learn the correct investigation steps

---

## SOC Alert Investigation Process

1. Monitor incoming alerts.
2. Prioritize alerts based on severity.
3. Take ownership of an alert.
4. Create an incident/case.
5. Follow the assigned playbook.
6. Gather evidence and investigate.
7. Decide whether the alert is a True Positive or False Positive.
8. Document findings.
9. Close the case.

---

## Responsibilities of a SOC Analyst

- Monitor alerts
- Investigate suspicious activity
- Follow playbooks
- Determine True Positive or False Positive
- Document findings
- Escalate incidents when necessary
- Provide feedback to improve SIEM detection rules

---

## Key Takeaways

- An **Event** is any activity occurring on a system or network.
- An **Alert** is a notification generated by a detection rule.
- An **Incident** is a confirmed or suspected security issue requiring action.
- Not every alert is an incident.
- Incident Management Systems are the primary workspace for SOC analysts.
- Threat Intelligence enriches investigations with additional context.
- SOAR automates repetitive response tasks.
- Playbooks ensure consistent and efficient investigations.
- Good documentation and case naming make future investigations easier.
