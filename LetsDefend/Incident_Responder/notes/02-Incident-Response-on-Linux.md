## Incident Response Lifecycle

### 1. Preparation
- Central log collection system
- NTP time synchronization across all devices
- Consistent user account naming conventions
- Document system/service administrators
- Asset management with patch versions
- Secure out-of-band communication methods
- Legal procedures documented in advance

### 2. Identification
- Gather preliminary information
- Assign primary investigator
- Use standardized checklists
- Determine if event is suspicious or false positive

### 3. Scope
- Characterize event type (DDoS, malware, data leak, etc.)
- Take immediate containment actions (account disable, IP block)
- Collect volatile memory, firewall logs, network traffic
- Isolate compromised systems (not just unplug)

### 4. Eradication
- Identify root cause
- Clean disk if rootkit suspected, restore from clean backup
- Apply latest updates after restoration
- Improve defenses based on findings
- Run vulnerability scans

### 5. Recovery
- Verify logging, systems, applications work correctly
- Coordinate restoration process
- Monitor for recurring events

### 6. Lessons Learned
- Document findings for both technical and executive audiences
- Identify what worked well and what failed
- Provide recommendations for improvement

## Incident Response Procedure (SOC Workflow)

**Alert** → **Analyze** → **Investigate** → **Assess Impact** → **Contain** → **Respond**

- **Alert:** Generated by SIEM from EDR, IDS, IPS, WAF logs
- **Analyze:** Tier 1 analyst determines false positive or real threat
- **Investigate:** Source of attack, attacker progress, timeline
- **Assess Impact:** Affected systems, damage assessment
- **Contain:** Isolate affected devices from network
- **Respond:** Remove threats, restore systems, apply lessons

## 3 Critical Questions for Any Investigation

1. Is there malware actively running in the system?
2. Is there suspicious internal or external communication?
3. Is there persistence established?

## Users and Groups

### Key Files

**/etc/passwd**
- Contains usernames, UID/GID, home directory, shell information
- Readable by all users
- Format: username:password:UID:GID:comment:home:shell
- If shell is /usr/sbin/nologin, user cannot login
- www-data user with nologin can still execute commands via web exploits

**/etc/shadow**
- Contains encrypted user passwords
- Only readable by root and shadow group
- Format: username:encrypted_password:last_change:min:max:warn:inactive:expire:reserved
- Passwords are hashed, not plaintext

**/etc/group**
- Contains groups and member users
- Readable by all users
- Format: group_name:password:GID:user_list

**/etc/sudoers**
- Defines sudo permissions
- Format: User_List Host_List = (Operator_List) Tag_List Command_List
- Use visudo to edit (syntax checking)

### Other Important Files
- /var/run/utmp: Current system status, active logins
- /var/log/wtmp: Historical login records
- /var/log/btmp: Failed login attempts

### Detection Commands

**List users:** `cat /etc/passwd`


**Check user creation history:** `grep useradd /var/log/auth.log`


**Check password changes:** `grep passwd /var/log/auth.log`


**Check group modifications:**
```bash
grep groupadd /var/log/auth.log
grep usermod /var/log/auth.log
```

**View login history:**
```bash
last
cat /var/log/wtmp
```

**View failed login attempts:**
```bash
grep "Failed password" /var/log/auth.log
journalctl _SYSTEMD_UNIT=ssh.service | egrep "Failed|Failure"
```
**List active users:**
```
w
who
users
```

### Critical Groups to Monitor
- root
- adm
- shadow
- sudo

### SSH User Detection
1. Read /etc/passwd
2. Remove users without valid shell
3. Remove users without valid passwords
4. Check /etc/ssh/sshd_config for AllowUsers

### Remediation Commands

**Delete user:** `userdel -r USERNAME`

**Remove user from group:** `gpasswd -d USERNAME GROUP`

**Edit sudoers:** `visudo`

**Change password:** `passwd USERNAME`

**Regenerate SSH keys:**
- Delete old keys
- Create new SSH key

## Processes

### Process States
- **Running:** Actively running or ready to run
- **Waiting:** Waiting for event or resource
- **Stopped:** Process is stopped
- **Zombie:** Process ended but task_struct not deleted

### Process Creation (fork & exec)
- **fork:** Creates child process by duplicating running process
- **exec:** Replaces process image with new image

### Key Concepts
- **PID:** Process Identifier (unique)
- **PPID:** Parent Process Identifier
- **init:** First process (PID = 1)
- /proc/sys/kernel/pid_max: Maximum PID value

### Detection Commands

**List processes:**
```bash
ps aux --forest
ps aux
ps -ef
```

**Process tree view:** `pstree`

**Real-time monitoring:** `top`


- Press P: Sort by CPU usage
- Press N: Sort by PID
- Press T: Sort by runtime

**Check specific process:** `ps aux | grep PROCESS_NAME`

### /proc Directory Analysis
- /proc/PID/status: Process status, user/group identity, PID/PPID
- /proc/PID/cmdline: Command line parameters
- /proc/PID/environ: Environment variables
- /proc/PID/fd: File descriptors

**Read command line:** `cat /proc/PID/cmdline`


### Suspicious Indicators
- Process names: shell, reverse, miner
- Service accounts running commands: www-data
- Binary location: /tmp/reverse
- Command line containing: password, login, URL, IP, port
- Default tools used maliciously (GTFOBins reference)

### Memory Analysis
`pmap PID`

### Remediation Commands

**Kill process:**
```bash
kill SIGNAL PID
kill -9 PID
```

**Kill all processes by name:** `killall PROCESS_NAME`

## Files and File System

### Linux Directories
- /boot: Boot files
- /opt: Optional software
- /etc: Configuration files
- /bin: Executable files
- /sbin: Superuser executables
- /var: Log files
- /dev: Device files
- /home: User home directories
- /root: Root home
- /usr: Applications and libraries

### Suspicious Directories to Check
- /tmp: Writable by all, files auto-delete
- /var/www: Web directories
- /var/log: Log files (attacker may alter)

### File Analysis Commands

**Find files by extension:**
```bash
find / -type f -iname \*.php -o -iname \*.php7 -o -iname \*.sh -o -iname \*.elf
```

**Find by modification time:**
```bash
find /tmp -newermt "2021-10-25 00:00:00" ! -newermt "2021-10-25 23:59:00"
find / -mtime +X
find / -mtime -X
```

**Find by owner:**
```bash
find /tmp -user www-data
```

**Find by change time:**
```bash
find / -ctime +X
```
**File metadata:**
```bash
stat FILENAME
```

### Suspicious File Indicators
- Suspicious extensions: .sh, .php, .php7, .elf
- Files in /tmp directory
- Files owned by compromised users
- Files modified during attack timeframe

### Remediation
- Delete attacker-written files
- Revert modified files to normal state
- Use clean image or snapshot if possible

## Services

### Service States
- **Active (running):** Service running in background
- **Active (exited):** Service ran but no daemon to monitor
- **Active (waiting):** Waiting for event
- **Inactive:** Not working
- **Enabled:** Starts at boot
- **Disabled:** Will not start at boot

### Service Configuration File Sections

**[Unit] Section**
- Description: Unit explanation
- Requires: Required other units
- Before: Units that must run before

**[Install] Section**
- WantedBy: How unit is activated

**[Service] Section**
- ExecStart: Commands to initiate service
- ExecStartPre: Commands before service starts
- Restart: Conditions for restart

### Detection Commands

**List all services:**
```bash
service --status-all
systemctl list-units --type=service
```

**Check service status:**
```bash
service SERVICE_NAME status
systemctl status SERVICE_NAME
```

**View service config:**
```bash
cat /lib/systemd/system/SERVICE_NAME.service
```

**View service logs:**
```bash
journalctl -u SERVICE_NAME
sudo journalctl -u cron --since "2021-05-07 00:00" --until "2021-05-07 23:59"
```

**View historical service activity:**
```bash
sudo journalctl --since "2021-05-07 00:00" --until "2021-05-07 23:59"
sudo journalctl -u cron --since "2021-05-07 00:00" --until "2021-05-07 23:59"
```

**Check config modification time:**
```bash
stat /lib/systemd/system/cron.service
```

### Suspicious Indicators
- Modified ExecStart directive
- Unknown services
- Services running from suspicious locations
- Services created during attack timeframe

### Remediation Commands

**Stop service:**
```bash
sudo systemctl stop SERVICE_NAME
sudo service SERVICE_NAME stop
```

**Disable service:**
```bash
sudo systemctl disable SERVICE_NAME
```

**Remove service files:**
```bash
rm FILE_NAME
```

**Reload systemd:**
```bash
sudo systemctl daemon-reload
```

## Crontab

### Key Concepts
- **Cron:** Tool for scheduling tasks
- **Cron job:** Timed job
- **Crontab:** Configuration file with cron jobs

### Crontab Format
Minute(0-59) Hour(0-23) Day(1-31) Month(1-12) DayOfWeek(0-6) Command

text

### Crontab Locations
- System crontab: /etc/crontab
- System drop-in: /etc/cron.d/*
- User crontabs: /var/spool/cron/crontabs/* or /var/spool/cron/*

### Detection Commands

**View system crontab:**
```bash
cat /etc/crontab
```

**View drop-in files:**
```bash
cat /etc/cron.d/*
```

**View user crontabs:**
```
cat /var/spool/cron/crontabs/*
cat /var/spool/cron/*
```

**View cron execution logs:**
```bash
cat /var/log/syslog | grep CRON
journalctl -u cron
journalctl -u cron --since "2021-05-07 00:00" --until "2021-05-07 23:59"
```

### Suspicious Indicators
- Bash scripts from /tmp directory
- Reverse shell commands
- Scripts named: backdoor, rev, shell
- New cron jobs during attack timeframe
- Commands in cron jobs from suspicious users

### Remediation Commands

**Edit crontab:**
```bash
crontab -e
crontab -u USERNAME -e
```

## SSH Authorized Keys

### Key Concepts
- SSH keys more secure than passwords
- Public key stored in ~/.ssh/authorized_keys
- Attackers add their own keys for persistence

### Detection Commands

**Find all authorized_keys files:**
```bash
find / -name 'authorized_keys'
```

### Suspicious Indicators
- New SSH keys added during attack timeframe
- Keys belonging to unknown users
- Modified authorized_keys files

### Remediation
- Remove attacker's SSH keys from authorized_keys files

## Bash_rc & Bash_profile

### Key Concepts
- **.bashrc:** Executed for non-login shells
- **.bash_profile:** Executed for login shells
- Attackers add reverse shell commands for persistence

### Detection Commands
```bash
cat .bashrc
cat .bash_profile
find / -name '.bashrc' -o -name '.bash_profile'
```

### Suspicious Indicators
- Reverse shell commands
- Malicious commands in .bashrc or .bash_profile
- Files modified during attack timeframe

### Remediation
- Remove malicious entries from .bashrc and .bash_profile

## Key Takeaways

- Everything is a file on Linux - analyze files for evidence
- Attackers use generic names: support, service, dev, admin, sysadmin
- Check both current state AND historical logs
- Persistence is priority - check all persistence points
- Default tools are weaponized (GTFOBins reference)
- /tmp directory is primary attack location
- Always run analysis as root when possible
- Time synchronization critical for correlation
- Use VirusTotal for quick file reputation checks
- Document everything for lessons learned
- Process tree analysis reveals malicious relationships
- No mount/unmount logs by default - check active mounts only
